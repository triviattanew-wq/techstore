2.4. Разработка серверной части

Серверная часть обрабатывает входящие HTTP-запросы, выполняет бизнес-логику,
обращается к базе данных и формирует ответы в формате JSON для клиента либо
готовый HTML при серверном рендеринге (SSR).

Ключевые элементы серверной части:

— lib/prisma.ts — единый экземпляр Prisma Client, используемый во всём приложении.
  Паттерн единственного экземпляра предотвращает исчерпание пула соединений
  с базой данных при горячей перезагрузке в режиме разработки.

— app/api/products/route.ts — эндпоинт выборки товаров с поддержкой фильтрации
  по категории, брендам, диапазону цены, объёму памяти, цвету, признакам
  (в наличии / новинка / хит), а также сортировки, пагинации и полнотекстового
  поиска.

— app/api/cart/route.ts — операции с корзиной: получение, добавление позиции,
  изменение количества, удаление. Корзина привязывается либо к авторизованному
  пользователю, либо к анонимной сессии через sessionId.

— app/api/admin/* — административные эндпоинты для управления товарами,
  категориями, брендами, заявками и просмотра статистики. Каждый обработчик
  проверяет наличие активной сессии с ролью ADMIN или MANAGER перед выполнением
  операции.

— middleware.ts — задаёт заголовки кэширования и ограничивает matcher для
  публичных и административных маршрутов.

Аутентификация и роли:

В схеме Prisma предусмотрено перечисление UserRole со значениями USER, ADMIN,
MANAGER и CONTENT_MANAGER. Для входа в административную панель используется
отдельная модель AdminUser. Управление сессиями реализовано через библиотеку
NextAuth с адаптером Prisma, что обеспечивает безопасное хранение токенов
и учётных данных в базе данных.

Безопасность:

— Пароли хранятся исключительно в виде хеша (поле passwordHash), вычисляемого
  алгоритмом bcrypt при регистрации или создании пользователя.
— Все административные API-маршруты защищены проверкой сессии и роли:
  при отсутствии прав возвращается статус 401 или 403.
— Входные данные форм и API-запросов проходят серверную валидацию с помощью
  библиотеки Zod, что исключает обработку некорректных или вредоносных данных.

(товары, категории, корзина, заявки, Trade-in, IMEI) и защищённые
административные маршруты (app/api/admin/*).

Публичные маршруты:
  GET  /api/products          — список товаров с фильтрами
  GET  /api/categories        — список категорий
  GET  /api/brands            — список брендов
  POST /api/cart/items        — добавить товар в корзину
  GET  /api/cart              — получить корзину
  POST /api/leads             — создать заявку (оформление заказа)
  POST /api/one-click         — заявка «купить в 1 клик»
  POST /api/trade-in          — заявка Trade-in
  POST /api/imei-check        — проверка IMEI
  POST /api/promo/check       — проверка промокода

Защищённые административные маршруты (требуют роль ADMIN/MANAGER):
  GET/POST        /api/admin/products       — список и создание товаров
  GET/PUT/DELETE  /api/admin/products/[id]  — редактирование и удаление товара
  GET/POST        /api/admin/categories     — управление категориями
  GET/POST        /api/admin/brands         — управление брендами
  GET             /api/admin/leads          — список заявок
  PATCH           /api/admin/leads/[id]/status — смена статуса заявки
  GET             /api/admin/stats          — статистика панели
------------------------------------------------------------

Таким образом, серверная часть приложения построена на принципах разделения
ответственности: каждый маршрут решает одну задачу, доступ к данным
централизован через Prisma, а безопасность обеспечивается на уровне
аутентификации, авторизации и валидации входных данных.
